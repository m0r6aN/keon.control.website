Keon Cortex  ·  Deterministic Memory Substrate

Vector databases
are not memory.

Most memory systems optimize for retrieval speed. Cortex optimizes for correctness — under restart, under failure, under multi-tenancy, under adversarial conditions. Ten non-negotiable invariants. Violations are bugs. There are no exceptions.

I1 — Deterministic IDs
Same inputs. Same shard. Every time.
shard_id = sha256(tenant_id + ":" + source + ":" + content + ":" + extractor_version)
No randomness. No timestamps. The same memory is always the same shard.
I2 — Fail-Closed Tenancy
Missing tenant → ValueError
Mismatched tenant → PermissionError. No silent pass-through. No cross-tenant bleed.
I3 — Idempotent Enqueue
Same key. One write. Always.
Same idempotency_key → subsequent enqueues are silent no-ops. Outbox: pending → leased → processed.
I4+I5 — Exclusive Lease + Recovery
Atomic. No two workers. Ever.
lease_next() is atomic. Expired leases recover automatically, preserving the attempt counter. No lost work.
I7 — Tenant Pre-Filter
Hard filter before the vector query.
A second post-filter catches any provider bug. Cross-tenant results are logged as errors and discarded. Two layers. Zero bleed.
I9+I10 — Decay Determinism + Bounded Scores
Pure function. Clamped output.
exponential_decay(initial, elapsed, half_life) — same inputs, same float. All composite scores clamped to [0.0, 1.0].

The four properties of governed execution
Every action. Every property. Enforced.

Policy-bound. Receipt-verified. Append-only. Fail-closed. These aren't aspirations — they're architectural invariants. Violations are bugs. There are no exceptions.

Policy-bound
Every action is evaluated against policy before it executes. No exceptions.
Receipt-verified
Every decision is cryptographically signed and cannot be altered after issuance.
Append-only
The ledger is immutable. Events are sequenced and sealed against reordering or deletion.
Fail-closed
On error or ambiguity, Keon denies — not permits. Safety is the default state.
